# System-Level Explainability (End-to-End)This document is the single coherent explanation of how one user turn executes from entrypoint to sealed completion.## End-to-End Path1. Public entrypoint- `DadBotTurnMixin.execute_turn(...)` receives a `TurnRequest` and routes sync/async delivery.2. Request normalization- `live_turn_request(...)` builds canonical input envelope (`mode`, `delivery`, `session_id`, timeout, constraints).3. Delegation to orchestration- `DadBotOrchestrator.handle_turn(...)` delegates to `bot.execute_turn(...)` when the orchestrator is the active execution authority.4. Control-plane submission- Orchestrator uses `ExecutionControlPlane.submit_turn(...)` via `_submit_turn_via_control_plane(...)`.- The control plane enforces scheduling, lease/lock policy, and bounded execution via `_execute_with_boundary(...)`.5. Boundary assertion at graph runtime- `TurnOrchestrationMixin.execute(...)` verifies required execution-boundary token before graph dispatch.6. Turn start instrumentation- Execution trace starts with `turn_start`.- Execution profile is normalized and attached to metadata.7. Recovery and capability security- `_resume_from_recovery(...)` checks durable resume point and restores completed stages.- `_apply_capability_security(...)` freezes capability set on fresh turns or verifies frozen set on resumed turns.8. Pipeline materialization- Pipeline stages are resolved by `_pipeline_items()`.- Contract version hash is stamped into determinism manifest.9. Stage execution loop- Topology provider invokes each stage through `_lg_node_executor(...)`.- `_run_stage(...)` applies stage-entry checks, node execution, contract checks, and checkpoint/receipt emission.10. Per-stage policy and mutation enforcement- `StageEntryGate` decides skip/entry and ordering.- `MutationGuard` wraps non-commit stage mutation boundaries.- Kernel decisions (`ok`, `error`, `rejected`) are converted to explicit trace events.11. Per-stage completion artifacts- `stage_done` trace event- before/after/edge checkpoints- durable recovery stage completion- signed receipt via side-effect adapter12. Turn branch resolution- Exception path -> `_handle_execute_exception(...)`- Short-circuit path -> `_finalize_short_circuit(...)`- Normal path -> `_finalize_normal_result(...)`13. Finalization seal- `_finalize_execution_trace_contract(...)`- `_seal_execution_identity(...)`- `TurnFinalizationInvariantGate.validate_turn_finalization(...)`14. Recovery cleanup- Successful finalized turns clear recovery records.15. Structural safety net- Outer `finally` calls `_execute_finalization_safety_net(...)` if any unexpected control path bypassed branch finalizers.## Explainability Outputs You Can Inspect- `state["execution_trace"]`: ordered event log for stage enter/skip/done/error and turn-level events.- `state["execution_trace_contract"]`: deterministic trace summary (`event_count`, `trace_hash`, version stamp).- `state["execution_identity"]`: sealed execution fingerprint.- `state["turn_finalization_invariant_report"]`: pass/fail plus concrete invariant violations.- `metadata["execution_failure"]` and `metadata["execution_failure_lifecycle"]` on failure paths.- capability audit report payload when audit mode is enabled.## Explainability ContractA turn is explainable if and only if these are all present at terminal state:1. explicit turn branch outcome (`turn_succeeded` | `turn_short_circuit` | `turn_failed`)2. contiguous execution-trace sequence with declared event types only3. sealed execution trace contract4. sealed execution identity5. finalization invariant reportIf any are missing, the execution must be treated as structurally incomplete.
