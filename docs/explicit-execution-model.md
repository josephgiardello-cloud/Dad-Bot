# Explicit Execution ModelStatus: normative runtime model for current Dad-Bot turn execution.## 1) Model StatementThe system execution model is:- behavior = validated turn state machine + enforced stage contracts + sealed finalization artifactsWhere:- Turn state machine authority lives in `TurnOrchestrationMixin.execute(...)`.- Stage admission authority lives in `StageEntryGate`.- Failure normalization authority lives in `ExecutionFailure` + `FailureStateMachine`.- Finalization authority lives in `TurnFinalizationInvariantGate` + execution trace contract + execution identity sealing.This replaces implicit "code + convention" behavior with explicit guards at runtime.## 2) Execution Entities- Request envelope: `TurnRequest` / `TurnResponse` (`dadbot/core/execution_contract.py`).- Turn carrier: `TurnContext` (`dadbot/core/graph_context.py`).- Dispatcher: `TurnOrchestrationMixin.execute(...)` (`dadbot/core/turn_orchestration_mixin.py`).- Policy engine: `ExecutionPolicyEngine` + `StageEntryGate` (`dadbot/core/execution_policy.py`).- Recovery coordinator: `ExecutionRecovery` (`dadbot/core/execution_recovery.py`).- Failure lifecycle: `FailureStateMachine` (`dadbot/core/failure_lifecycle.py`).- Finalization gate: `TurnFinalizationInvariantGate` (`dadbot/core/invariant_gate.py`).## 3) Formal Turn State MachineStates (runtime):1. `TURN_START`2. `PREPARED` (profile normalized, boundary validated)3. `RESUME_CHECKED`4. `CAPABILITY_FROZEN_OR_VERIFIED`5. `STAGE_LOOP`6. `SHORT_CIRCUITED` or `FAILED` or `SUCCEEDED`7. `FINALIZED` (trace contract + identity + finalization gate)Allowed transition skeleton:1. `TURN_START -> PREPARED`2. `PREPARED -> RESUME_CHECKED`3. `RESUME_CHECKED -> CAPABILITY_FROZEN_OR_VERIFIED`4. `CAPABILITY_FROZEN_OR_VERIFIED -> STAGE_LOOP`5. `STAGE_LOOP -> SHORT_CIRCUITED | FAILED | SUCCEEDED`6. `SHORT_CIRCUITED | FAILED | SUCCEEDED -> FINALIZED`No bypass path may skip `FINALIZED` because `execute(...)` has an outer safety net (`_execute_finalization_safety_net`).## 4) Stage Admission ModelEach stage must pass all checks before execution:1. Recovery skip check (`check_recovery_skip`).2. Capability skip check (`check_capability_skip`).3. Stage ordering enforcement (`enforce_stage_ordering`).4. Input contract check (`_enforce_node_stage_contract(..., phase="input")`).5. State schema mutation check (`_enforce_stage_state_schema`).6. Output contract check (unless runtime-kernel rejection path applies).If a check fails, execution emits an explicit stage event (`stage_skip`, `stage_error`, `kernel_rejected`) and follows failure/finalization handling.## 5) Determinism and Contract SealingThe model explicitly seals deterministic artifacts:- `execution_trace_contract` hash is finalized at turn end.- `execution_identity` fingerprint is sealed at turn end.- Optional expected-hash comparison can hard-fail if trace hash diverges.Canonicalization removes volatile timing detail from contract hashing, so determinism focuses on semantic ordering and event content, not wall-clock jitter.## 6) Completion Invariants (Turn-Level)Turn completion is accepted only if invariant validation passes:- required stage coverage (when `require_complete=True`)- phase-order monotonicity- checkpoint chain integrity- tool plan/execution completeness- ledger transaction integrityViolation raises `InvariantViolationError` and blocks successful sealing.## 7) Runtime GuaranteesGiven this model, a completed turn guarantees:1. ordered, contiguous execution trace events2. explicit admission decisions for each stage (enter/skip/error/rejection)3. normalized failure taxonomy on failure paths4. sealed execution identity and trace contract artifacts5. finalization invariant report persisted in turn state
